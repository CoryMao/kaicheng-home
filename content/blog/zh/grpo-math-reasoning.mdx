      <p>Reward 函数是 GRPO 训练最核心的组件，决定了模型优化的方向。我经历了 3 个主要版本的设计迭代，逐步收敛到稳定方案。</p>

      <p>第一版 Reward 如下：</p>
      <ul>
        <li>正确答案: +1.0；错误: −0.1</li>

        <li>不检查 Think 格式，纯答案驱动，减少 Reward 信号噪声</li>
      </ul>

      <h3>v1 —— 加 FP 惩罚：不能让模型无脑输出None</h3>
      <p>模型学会了 None 判别后出现副作用：在<strong>有解</strong>题目上也说 None（False Positive），大大拉低了 Overall 准确率。</p>
      <p><strong>修复</strong>: 对于 Ground Truth 不是 None 的题目，模型输出 None → <strong>扣 0.8</strong>（最严厉的单条惩罚）。</p>
      <p><img src="/media/blog/grpo-math-reasoning/GRPO Base-5epFP率.png" alt="FP 惩罚效果" /></p>

      <h3>v2 —— 加怪字惩罚（Unicode 噪声清除）</h3>
      <p>GRPO 训练中出现了独特现象：模型学会在答案中插入大量非 ASCII 乱码（如私用区字符、全角符号），推测是为了"碰巧"匹配 Ground Truth 而 Reward Hacking。</p>
      <p>我设计了一个精密的白名单机制：允许 ASCII + 希腊字母 + 数学运算符 + 箭头 + 上下标等 Unicode 范围，每个不在白名单内的字符扣 0.01。</p>
      <p><strong>效果</strong>: 怪字率从 43.5% <strong>降至 0%</strong>。</p>

        <em>惩罚后：输出结果正常</em>
      </p>

      <h2>个人反思</h2>

      <h3>技术反思</h3>
      <ol>
        <li>
          <strong>SFT 的局限性</strong>: SFT 能让模型去拟合训练数据的分布，对于Qwen3-0.6B这样参数量非常小的模型来说，单纯的 SFT 很难带来实质性的能力提升，甚至可能因为数据分布的改变而导致性能下降。在实验的最初期，我还没意识到这一点，所以一直提升不了正确率，确实是非常upset的。所以这个经验告诉我，SFT之后直接greedy解码出来的结果没有那么好，并不一定是训练没做好，而是需要转向RL来提高模型的采样效率。
        </li>
        <li>
          <strong>RL的直觉</strong>: 做RL的先决条件是模型的采样分布中已经包含正确答案（即 Pass@k 已经远高于 Greedy）。如果模型连正确答案都采样不出来，RL 就无从优化。这个经验判断与 RLVR 相关研究中"base model 的大 $k$ pass@k 可以视为当前 RL 可利用能力上界"的结论一致。而GRPO的核心则在于设计一个合理的Reward函数（这个必须要通过试错才能找到），来提高模型的采样效率。
        </li>
      </ol>

      <h3>项目管理反思</h3>
      <ol>
        <li><strong>数据质量的重要性</strong>: 训练数据的质量不仅包括其正确性，还有其类型的分布是否均衡，是否包含模型需要学习的能力点（如 None 判别）。</li>
        <li><strong>如何使得团队成员的效率更高</strong>：下达指令时，不能假定他们理解自己的观点，因为实验是我一个人做的。必须给出我的观点的依据，解释清楚，告诉他们接下来要做什么、具体的步骤（最好是few-shot），这一点我觉得跟提高模型表现很相似。</li>
      </ol>
